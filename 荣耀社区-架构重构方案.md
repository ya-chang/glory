## 一、现状分析

### 当前架构

```
┌─────────────────┐      ┌──────────────────┐      ┌─────────────┐
│   GitHub Pages   │─────▶│  阿里云 FC (Express) │─────▶│  OSS (JSON)  │
│   静态 HTML/CSS  │      │    后端 API        │      │  glory-db.json│
│   17 个 HTML 文件 │      │    1126 行 app.js  │      │  全量读写     │
└─────────────────┘      └──────────────────┘      └─────────────┘
```

### 代码规模

| 文件类型 | 文件数 | 总行数 |
|---------|--------|--------|
| HTML 页面 | 17 | ~3,500 |
| CSS 样式 | 2 | ~1,800 |
| 前端 JS | 5 | ~2,400 |
| 后端 JS | 4 | ~1,900 |
| **合计** | **28** | **~9,600** |

### 当前技术栈

- **前端**：原生 HTML + CSS + JavaScript，无框架，无构建工具
- **后端**：Express.js，部署在阿里云函数计算 (FC)
- **数据库**：单个 JSON 文件存储在阿里云 OSS
- **部署**：GitHub Pages（前端）+ 阿里云 FC（后端 API）
- **认证**：JWT + localStorage
- **图片**：第三方图床 (freeimage.host)

## 二、核心问题

### 🔴 严重（必须修复）

**1. JSON 文件数据库**

```
问题描述：
- 每次请求全量加载 JSON 到内存
- 写入使用 500ms 防抖，并发写会丢数据
- 无索引，搜索是全表遍历 O(n)
- 数据量增长后内存占用线性上升

影响：
- 100 个帖子时感觉不到问题
- 1000 个帖子时 API 响应变慢
- 并发写入时数据丢失（两个用户同时发帖/回复）
```

**2. 前端代码大量重复**

```
问题描述：
- 17 个 HTML 文件复制粘贴相同的导航栏、页脚
- 改一个导航链接需要修改 17 个文件
- 每个页面内联 100-300 行 JavaScript
- 同一种样式在不同页面写法不一致

影响：
- 维护成本高，容易漏改
- 帖子卡片样式改了列表页，搜索页还是旧的
- 新增页面要复制大量模板代码
```

### 🟡 中等（应该修复）

**3. 前端无错误边界**

```
问题描述：
- API 调用失败时大部分页面只是 console.error
- 用户看到的是白屏或永远"加载中"
- 没有统一的错误处理和重试机制

影响：
- 网络波动时用户体验极差
- 难以定位前端错误
```

**4. CSS 无设计系统**

```
问题描述：
- style.css 39KB 全是手写样式
- 同一种蓝色有多种写法（#1a73e8, #4a90d9, var(--blue-3)）
- 按钮、卡片、输入框的样式散落各处
- 大量内联 style="..." 在 HTML 中

影响：
- 视觉不一致
- 新增组件时不知道该用哪个变量
- 修改全局样式容易遗漏
```

### 🟢 低（可以优化）

**5. 无构建工具**

```
- 无代码压缩、无 tree-shaking
- CSS/JS 直接加载，无分包
- 无法使用 TypeScript、ES6+ 新语法
```

**6. 无 SEO 优化**

```
- 单页应用不利于搜索引擎抓取
- 无 sitemap.xml
- 无结构化数据
```

**7. 无 PWA 支持**

```
- 不支持离线浏览
- 无安装到桌面的能力
- 无推送通知
```

## 三、重构目标

### 核心原则

> **用最小的工程量，获得最好的可维护性。**
> 不追潮流，只追"改得动、不出事"。

### 量化目标

| 指标 | 现在 | 目标 |
|------|------|------|
| 新增页面所需时间 | 2-3 小时（复制模板） | 15 分钟（新建组件） |
| 改导航栏需改文件数 | 17 → 1（layout.js） | 1（AppHeader.vue） |
| API 错误时用户感知 | 白屏 | 错误提示 + 重试按钮 |
| 并发写入安全性 | ❌ 会丢数据 | ✅ 数据库事务保证 |
| 搜索性能 | O(n) 全表扫描 | O(log n) 索引查询 |
| 首屏加载时间 | 3-5 秒 | < 1 秒 |

## 四、技术选型

### 方案对比

| 维度 | 方案 A：Vue + D1 | 方案 B：Vue + Supabase | 方案 C：Alpine.js + D1 |
|------|-----------------|----------------------|----------------------|
| 前端框架 | Vue 3 + Vite | Vue 3 + Vite | Alpine.js |
| 后端 | Cloudflare Workers | Supabase Edge Functions | Cloudflare Workers |
| 数据库 | Cloudflare D1 (SQLite) | Supabase PostgreSQL | Cloudflare D1 (SQLite) |
| 认证 | 自建 JWT | Supabase Auth（开箱即用） | 自建 JWT |
| 文件存储 | Cloudflare R2 | Supabase Storage | Cloudflare R2 |
| 学习成本 | 中等 | 低 | 低 |
| 灵活性 | 高 | 中 | 高 |
| 免费额度 | 10 万次/天 | 5 万月活 | 10 万次/天 |
| 适合场景 | 需要完全控制 | 快速开发 | 轻量改造 |

### 推荐方案：方案 A（Vue 3 + Cloudflare Workers + D1）

**推荐理由：**

1. **Cloudflare D1** — 免费 SQLite，有索引，并发安全，全球边缘节点
2. **Cloudflare Workers** — 免费 10 万次/天，冷启动 < 1ms，比阿里云 FC 快
3. **Vue 3 + Vite** — 组件化开发，响应式数据，生态成熟，国内社区大
4. **Cloudflare R2** — 免费 10GB 存储，替代第三方图床

**免费额度是否够用：**

```
假设社区规模：
- 500 注册用户
- 日活 100 人
- 每人每天浏览 20 页，发 1 帖 + 3 回复

每日 API 调用量：
- 页面加载：100 × 20 = 2,000 次
- 发帖/回帖：100 × 4 = 400 次
- 点赞/其他：200 次
- 合计：~2,600 次/天

Cloudflare Workers 免费额度：100,000 次/天
D1 免费额度：100,000 次读 + 50,000 次写/天

结论：免费额度足够，余量充裕
```

```sql
-- ======================
-- 用户表
-- ======================
CREATE TABLE users (
  email         TEXT PRIMARY KEY,
  username      TEXT NOT NULL,
  password_hash TEXT NOT NULL,
  role          TEXT NOT NULL DEFAULT 'user',  -- 'user' | 'admin'
  email_verified INTEGER NOT NULL DEFAULT 0,
  created_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 用户资料（扩展信息）
CREATE TABLE user_profiles (
  email       TEXT PRIMARY KEY REFERENCES users(email),
  avatar      TEXT DEFAULT '',
  bio         TEXT DEFAULT '',
  fav_player  TEXT DEFAULT '',
  fav_team    TEXT DEFAULT '',
  join_date   TEXT DEFAULT (date('now')),
  exp         INTEGER DEFAULT 0,
  level       INTEGER DEFAULT 1,
  checkin_streak INTEGER DEFAULT 0,
  checkin_total  INTEGER DEFAULT 0,
  checkin_last   TEXT
);

-- ======================
-- 帖子表
-- ======================
CREATE TABLE posts (
  id            TEXT PRIMARY KEY,
  title         TEXT NOT NULL,
  content       TEXT NOT NULL,
  category      TEXT NOT NULL,             -- original/fanwork/casual/meta
  author_email  TEXT NOT NULL,
  author_name   TEXT NOT NULL,
  author_level  INTEGER NOT NULL DEFAULT 1,
  is_pinned     INTEGER NOT NULL DEFAULT 0,
  is_locked     INTEGER NOT NULL DEFAULT 0,
  like_count    INTEGER NOT NULL DEFAULT 0,
  reply_count   INTEGER NOT NULL DEFAULT 0,
  view_count    INTEGER NOT NULL DEFAULT 0,
  tags          TEXT NOT NULL DEFAULT '[]', -- JSON array
  created_at    TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at    TEXT NOT NULL DEFAULT (datetime('now')),
  last_reply_at TEXT
);

-- 索引：按分类 + 最新回复排序（论坛列表页主查询）
CREATE INDEX idx_posts_category_time
  ON posts(category, last_reply_at DESC);

-- 索引：按作者查询（个人主页）
CREATE INDEX idx_posts_author
  ON posts(author_email, created_at DESC);

-- 索引：全局最新帖子
CREATE INDEX idx_posts_latest
  ON posts(created_at DESC);

-- ======================
-- 回复表
-- ======================
CREATE TABLE replies (
  id            TEXT PRIMARY KEY,
  post_id       TEXT NOT NULL,
  content       TEXT NOT NULL,
  author_email  TEXT NOT NULL,
  author_name   TEXT NOT NULL,
  author_level  INTEGER NOT NULL DEFAULT 1,
  reply_to      TEXT,                     -- 父回复 ID（楼中楼）
  mention_user  TEXT,
  like_count    INTEGER NOT NULL DEFAULT 0,
  created_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 索引：按帖子查询回复（帖子详情页主查询）
CREATE INDEX idx_replies_post
  ON replies(post_id, created_at);

-- ======================
-- 点赞表
-- ======================
CREATE TABLE likes (
  user_email  TEXT NOT NULL,
  target_type TEXT NOT NULL,               -- 'post' | 'reply'
  target_id   TEXT NOT NULL,
  created_at  TEXT NOT NULL DEFAULT (datetime('now')),
  PRIMARY KEY (user_email, target_type, target_id)
);

-- ======================
-- OC 角色表
-- ======================
CREATE TABLE ocs (
  id            TEXT PRIMARY KEY,
  name          TEXT NOT NULL,
  game_id       TEXT NOT NULL,
  avatar        TEXT DEFAULT '',
  illustration  TEXT DEFAULT '',
  age           TEXT,
  gender        TEXT,
  height        TEXT DEFAULT '',
  birthday      TEXT DEFAULT '',
  appearance    TEXT,
  personality   TEXT,
  class         TEXT NOT NULL,
  weapon        TEXT,
  weapon_type   TEXT DEFAULT '银武',
  weapon_desc   TEXT DEFAULT '',
  combat_style  TEXT,
  signature_skills TEXT DEFAULT '[]',
  custom_skill  TEXT DEFAULT '',
  team          TEXT,
  team_role     TEXT DEFAULT '队员',
  bio           TEXT DEFAULT '',
  relationships TEXT DEFAULT '[]',
  creator_email TEXT NOT NULL,
  creator_name  TEXT DEFAULT '',
  status        TEXT NOT NULL DEFAULT 'active',
  created_at    TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at    TEXT NOT NULL DEFAULT (datetime('now'))

CREATE INDEX idx_ocs_creator ON ocs(creator_email);
CREATE INDEX idx_ocs_class ON ocs(class, status);

-- ======================
-- 通知表
-- ======================
CREATE TABLE notifications (
  id          TEXT PRIMARY KEY,
  email       TEXT NOT NULL,
  type        TEXT NOT NULL,               -- 'reply' | 'like' | 'mention' | 'system'
  is_read     INTEGER NOT NULL DEFAULT 0,
  from_user   TEXT,
  post_id     TEXT,
  post_title  TEXT,
  content     TEXT DEFAULT '',
  created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_notif_email
  ON notifications(email, is_read, created_at DESC);

-- ======================
-- Token 表（验证/重置密码）
-- ======================
CREATE TABLE tokens (
  key        TEXT PRIMARY KEY,              -- 'type:token'
  email      TEXT NOT NULL,
  expires_at INTEGER NOT NULL
);
```

### 迁移脚本（JSON → SQL）

```js
// migrate.js - 一次性迁移脚本
const db = require('./glory-db.json'); // 旧数据

function migrate() {
  const sql = [];

  // 用户
  for (const [email, u] of Object.entries(db.users)) {
    sql.push(`INSERT INTO users VALUES('${esc(email)}','${esc(u.username)}','${esc(u.password)}','${u.role||'user'}',${u.emailVerified?1:0},'${u.createdAt||new Date().toISOString()}')`);
  }

  // 帖子
  for (const p of db.forumPosts) {
    sql.push(`INSERT INTO posts VALUES('${p.id}','${esc(p.title)}','${esc(p.content)}','${p.category}','${p.authorId}','${esc(p.authorName)}',${p.authorLevel||1},${p.isPinned?1:0},${p.isLocked?1:0},${p.likeCount||0},${p.replyCount||0},${p.viewCount||0}','${JSON.stringify(p.tags||[])}','${p.createdAt}','${p.updatedAt||p.createdAt}','${p.lastReplyAt||p.createdAt}')`)
  }

  // ... 回复、点赞、OC 同理
  return sql.join(';\n') + ';';
}
```

### Composables（组合式函数）

```js
// composables/useAuth.js
export function useAuth() {
  const user = ref(null);
  const token = ref(localStorage.getItem('glory_token'));

  async function login(email, password) { ... }
  async function register(email, username, password) { ... }
  function logout() { ... }
  function isLoggedIn() { ... }
  function isAdmin() { ... }

  return { user, token, login, register, logout, isLoggedIn, isAdmin };
}

// composables/useApi.js
export function useApi() {
  const loading = ref(false);
  const error = ref(null);

  async function get(path) {
    loading.value = true;
    error.value = null;
    try {
      const res = await fetch(API_BASE + path, { headers: authHeaders() });
      if (!res.ok) throw new Error((await res.json()).error);
      return await res.json();
    } catch (e) {
      error.value = e.message;
      throw e;
    } finally {
      loading.value = false;
    }
  }

  // post, put, delete 同理
  return { loading, error, get, post, put, del };
}

// composables/useToast.js
export function useToast() {
  const toasts = ref([]);

  function show(message, type = 'info', duration = 2500) {
    const id = Date.now();
    toasts.value.push({ id, message, type });
    setTimeout(() => {
      toasts.value = toasts.value.filter(t => t.id !== id);
    }, duration);
  }

  return { toasts, show };
}
```

## 八、迁移方案

### 阶段一：数据库迁移（1-2 周）

> **目标**：把 JSON 文件换成 SQLite，消除数据安全风险

```
步骤：
1. 在 Cloudflare 创建 D1 数据库
2. 执行 schema.sql 建表
3. 编写 migrate.js 把 glory-db.json 数据导入 D1
4. 后端 API 改为 SQL 查询（保持接口不变）
5. 前端代码不动，只换后端
6. 灰度切换：先让部分请求走新后端

风险：
- 数据迁移可能有格式问题，需要验证
- D1 有请求体大小限制（1MB），需要检查

收益：
✅ 并发写入不再丢数据
✅ 查询性能从 O(n) 变 O(log n)
✅ 可以加全文搜索（FTS5）
✅ 数据有持久化保障
```

### 阶段二：前端框架化（2-3 周）

> **目标**：用 Vue 3 重写前端，消除代码重复

```
步骤：
1. Vite + Vue 3 项目初始化
2. 搭建路由和基础组件（AppHeader、AppFooter、PostCard）
3. 迁移核心页面：
   - 首页 → Home.vue
   - 论坛列表 → ForumHome.vue + ForumCategory.vue
   - 帖子详情 → PostDetail.vue
   - 发帖 → PostNew.vue
4. 迁移其他页面
5. 迁移静态数据（选手、战队）
6. 测试 + 上线

风险：
- 样式可能有细微差异，需要像素级对比
- 旧的 localStorage 数据需要兼容

收益：
✅ 组件复用，改一处生效全局
✅ 路由切换无刷新，体验流畅
✅ 自动代码分包，加载更快
✅ 可以用 TypeScript（可选）
```

### 阶段三：体验优化（1 周）

> **目标**：提升用户体验

```
1. 骨架屏加载（替代"加载中..."文字）
2. 图片懒加载 + 渐进式加载
3. 深色模式（CSS 变量切换）
4. PWA 支持（Service Worker 离线缓存）
5. 移动端手势优化
6. 搜索优化（防抖 + 历史记录 + 热词）
```
